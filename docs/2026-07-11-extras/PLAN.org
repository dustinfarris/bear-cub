# Out-of-scope discoveries land here during implementation instead of being fixed inline. Promoted to v2 candidates at initiative close.

- From Story 01: a full =mix test= run occasionally logs a non-fatal =Exqlite.Connection ... disconnected: ** (DBConnection.ConnectionError) client ... exited= line during teardown (no test failures either way). Traced to the scratch =BearCub.MigrationTestRepo= pool (=test/support/migration_test_repo.ex=), started per-test via =start_supervised!= with the default (non-Sandbox) pool, shutting down at test-process exit — the same class of benign teardown race the MVP chain's PLAN.org already noted for Story 03 of that initiative. Never affected =mix precommit='s exit code across many repeated runs; worth a look only if fully log-clean =mix precommit= output becomes a goal.
- From Story 01: the DDL-transaction/connection-pool hazard the nullability migration works around — =Ecto.Migration.Runner= dispatches =execute=/=create table=/=drop table= through a separate Agent process, so with =@disable_ddl_transaction true= a pooled (non-Sandbox) connection can schedule statements on different physical SQLite connections, letting a later statement observe a stale pre-drop schema — was promoted directly to =docs/learnings.org= (2026-07-12 entry) rather than waiting for a second occurrence, per human ruling: transferable craft knowledge proven by reproduction on two paths (an isolated test repo and the real =mix ecto.migrate= path), not design-silence material. No further action pending here.
- From Story 02: =list_extras/2= (=lib/bear_cub/chores.ex=) references =BearCub.Chores.Completion= fully-qualified rather than the =alias BearCub.Chores.Completion= declared later in the file — correct given Elixir's lexical alias scoping (the alias isn't in scope yet at that point in the file), but a future pass could hoist the alias earlier for consistency.
- From Story 02: =update_chore/2= builds a partial =%Kid{id: chore.kid_id}= struct purely to satisfy =next_position/2='s =%Kid{}= guard clause, avoiding an extra query. Works correctly and is covered by tests, but is a slightly unusual pattern — a future pass could change =next_position/2='s signature to take a bare =kid_id= instead.
- From Story 03: =ChoreLive.Index='s three list sections (Morning/Evening/Extras) render plain =:for= lists rather than LiveView streams, per CLAUDE.md's "always use streams" guideline. Pre-existing pattern — Morning/Evening already did this before Story 03; the new Extras section just mirrors it. Consistent with the rest of the admin/kiosk codebase at family-scale list sizes (single-digit item counts); revisit only if that scale assumption changes.
