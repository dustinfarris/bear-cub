** Admin — fail-on-inspection (Screens)

The parent reviews a kid's day (phone-first, over Tailscale) and fails a chore the child faked. A new =fail_chore/2= context action stamps =undone_at= + =failed_at= on the current live completion; the existing ="chores"= PubSub broadcast drives the kiosk live update (chore reverts to incomplete, routine bonus drops if it had been complete, failed-chore UI appears, points badge and completion badge update). This is a distinct action from the existing on-behalf undo. (SC-3, D40)

Two guards protect the admin surface against a mis-tap, because a fail is /effectively irreversible/ — both =undone_at= and =failed_at= are stamped, there is no unfail action, and the only netting paths are the kid's in-window redo or the D49 parent on-behalf override (D53):

- *Confirmation before it runs* — tapping fail does not immediately invoke =fail_chore/2=; it first requires explicit confirmation. Declining leaves the completion untouched. The confirmation guards the permanent, punitive judgment against an accidental tap. =fail_chore/2= itself is unchanged — only the interaction gains a confirm step.
- *Solid, non-interactive flag after a fail* — once a chore is failed, its admin row shows the fail flag rendered *solid and non-interactive* for the rest of the day: standing evidence that the fail happened, visually distinct from the *actionable* fail flag offered on a still-done chore. This restores information the shipped one-tap surface lost, where the control simply vanished the moment the chore reverted to incomplete. (SC-3, SC-4, D53)

** Context (=BearCub.Chores=) — unchanged in this story

- =fail_chore/2= =(chore, local_now)= — stamp =undone_at= + =failed_at= on the current live completion; broadcast ="chores"=. Returns an error if there is no live completion to fail. The write path shipped in Story 05 is *not* touched — this story adds only the confirmation interaction and the standing flag on the admin surface.

** Decision Log entry in scope

- *D53 (2026-07-17) — Admin fail is confirmation-guarded; a failed chore's admin row keeps a solid, non-interactive fail flag for the day:* tapping fail in admin no longer runs =fail_chore/2= immediately — it requires explicit confirmation first, and after a fail the chore's row shows its fail flag rendered *solid and non-interactive* for the rest of the day, distinct from the *actionable* fail flag offered on a still-done chore. Rationale: a fail is effectively irreversible (both =undone_at= and =failed_at= are stamped, there is no unfail action; the only netting paths are the kid's in-window redo or the D49 parent on-behalf override) and punitive, so the confirmation guards a permanent judgment against a mis-tap, and the solid flag gives the admin standing evidence of the fail — where the shipped Story 05 control simply vanished the moment the chore reverted to incomplete. Rejected — immediate one-tap fail (the shipped Story 05 behavior): too easy to trigger a permanent penalty by an accidental tap. Rejected — a reversible fail/unfail toggle (considered and dropped by the human): it would erase penalty records and contradict SC-4's audit-trail permanence, where a fail row must persist forever. Settles SC-3 admin interaction safety; reinforces SC-4. → Screens (admin fail-on-inspection), Overview.

* User Story

As a parent, I need failing a chore to ask me to confirm before it commits — and to leave a standing mark on the failed chore for the rest of the day — so that I never impose a permanent, irreversible penalty by an accidental tap, and so I can see at a glance which chores I already failed today.

* Technical Notes

The Design excerpt above is the source of truth; DESIGN.org is broader context.

- Depends on Story 05 (=fail_chore/2= and the =#fail-chore-{id}= admin control it shipped). This story changes only the admin interaction and the failed-row rendering — the context write path, the PubSub broadcast, and the kiosk are untouched.
- Phoenix ~data-confirm~ (=phx-click= with a =data-confirm= attribute) is the expected shape for the confirmation guard — no bespoke modal LiveComponent. Confirm/decline is a client-side gate on the existing =phx-click="fail-chore"= handler; declining never reaches the server, so no completion is stamped.
- Story 05's fail tests are updated to drive *through* the confirmation: LiveView tests trigger the =fail-chore= event as the confirmed path (=render_click= on =#fail-chore-{id}= exercises the post-confirm handler), and add coverage that the control carries the =data-confirm= attribute. Do not delete Story 05's write-path assertions — extend them.
- The standing failed flag: render a *solid, non-interactive* fail marker (a stable DOM id, e.g. =#failed-flag-{chore.id}=, distinct from the =#fail-chore-{id}= actionable button) on a chore whose current-day state is failed-and-not-redone. "For the rest of the day" is the day-scoped =failed_at= fact — reuse the derivation already present (=failed_chore_ids/1= from Story 06), no new timestamp machinery. A row that is *done* (redoable) still shows the actionable fail button; a row that is *failed* shows the solid flag.
- Kiosk is untouched — the kiosk/admin fence is unaffected; this story lives entirely in =Admin.TodayLive=.
