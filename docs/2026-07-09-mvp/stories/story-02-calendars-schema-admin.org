- *Calendars are database data (D9)*, entered and edited through admin — one or more per kid, plus family calendars as =kid_id: null= (rendered in both columns). There is no local-secrets mechanism and no hand-placed /application or family/ secret on the box (infrastructure credentials for backups are a distinct, explicit rebuild input — §9). Rotating a URL is a phone action; every backup contains the secret URLs, so the off-box backup destination must be private — it already held kid names and the family schedule, so this tightens nothing that wasn't already required, but it is now load-bearing.
- *Integrity:*
  - =ics_url= uses Ecto's =redact: true=, so by construction the URL is omitted from inspected structs and renders as =**redacted**= in inspected changesets and crash reports.
- *Contexts:* =BearCub.Chores= (kids, chores, completions — pure local CRUD, no network) and =BearCub.Calendars= (calendar CRUD, fetching, parsing — the external integration). They share nothing, fail differently, and never call each other; only the web layer reads both. Calendar trouble structurally cannot touch the tap-to-complete path (FR-20).

* User Story

As a parent, I need to add, edit, and delete each kid's and the family's calendar ICS URLs from the admin UI on my phone, so that the calendars driving the kiosk's events strip are entirely data I control, with no repo or server file to touch.

* Acceptance Criteria

** [ ] From =/admin=, a parent can add a calendar with a label and an ICS URL, either assigned to a specific kid or marked as a family calendar (=kid_id: null=).
** [ ] From =/admin=, a parent can edit and delete an existing calendar's label or ICS URL.
** [ ] The calendars admin screen is reachable and usable from a phone browser, consistent with the rest of =/admin= (FR-25).
** [ ] The calendar schema's =ics_url= field is protected by Ecto's =redact: true=, verified by a test that inspects a calendar changeset or struct and asserts the URL does not appear in its inspected output.
** [ ] LiveView integration tests cover the create, edit, and delete flows for calendars (Repo conventions: §11 testing expectations fold into ACs wherever a flow is touched).

* Technical Notes

- Scaffold with =mix phx.gen.live= per the Design excerpt's Implementation directive (context, schema, migration, =Admin.CalendarsLive.Index/Form=, PubSub wiring), then reshape toward this design rather than hand-rolling.
- This story covers only calendar CRUD (data entry) — no fetching, no parsing, no ETS, no kiosk rendering. The Refresher GenServer that fetches =ics_url= and populates =last_payload=/=last_fetched_at=/=last_error= is Story 03, which depends on this story's schema and context existing. The kiosk events strip that renders parsed events is Story 04, which depends on Story 03.
- This story also does not depend on Story 01 (=BearCub.Calendars.ICS=) — CRUD on calendar rows doesn't parse anything. Both stories can be built in either order; D16 fixes Story 01 first only because it is the project's largest unknown, not because of a code dependency.
- Per §1's contexts paragraph, =BearCub.Calendars= must not be called by =BearCub.Chores=, and vice versa — this story is establishing the =BearCub.Calendars= context boundary that Story 03 and 04 build on.
- The migration must match the schema fields verbatim from the Design excerpt above: =kid_id= (nullable, references =kids=), =label=, =ics_url= (=redact: true=), =last_payload=, =last_fetched_at=, =last_error=, timestamps. =last_payload=/=last_fetched_at=/=last_error= are written by the Refresher in Story 03 — this story only needs the columns to exist, not populated.
