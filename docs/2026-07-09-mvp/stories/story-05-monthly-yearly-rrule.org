#+title: Story 05 — Monthly RRULE expansion & graceful degradation on unsupported FREQ

Design: [[file:../DESIGN.org][DESIGN.org]] · Plan: [[file:../PLAN.org][PLAN.org]]

* Design excerpt

Verbatim from DESIGN.org §6 (Calendar pipeline):

#+begin_quote
*Parse:* at fetch time (and once on boot from cached payloads), payloads are parsed and recurring events expanded into concrete instances over a bounded horizon (yesterday → tomorrow, so midnight-spanning and recurring events resolve), held in ETS. Parsing sits behind our own =Calendars.ICS= module: hand-rolled (D25) — the =icalendar= hex package silently drops =RECURRENCE-ID=, an unrecoverable gap against the correctness bar below — fixture-tested against sanitized excerpts of the family's real Google Calendar ICS exports, and swappable behind the wrapper.
#+end_quote

Verbatim from DESIGN.org §11 (Testing strategy):

#+begin_quote
*Unit:* =Routines= window/boundary edges exhaustively (23:59/00:01, edge-to-edge handoff, overnight gap); completion semantics (complete→undo→complete AC, double-tap uniqueness via the partial index, local-date scoping across midnight); =Calendars.ICS= against sanitized fixtures derived from real Google ICS exports (all-day, timed, RRULE recurrence, EXDATE exclusion, RECURRENCE-ID override, midnight-spanning); seed idempotency.
#+end_quote

Verbatim from PRD.org (FR-19):

#+begin_quote
*FR-19* Each kid's column shows their personal events blended with family events in one chronological list; family events appear in both columns and are visually distinct (family color/icon). Today's events only — no tomorrow preview.
#+end_quote

* User Story

As a parent relying on the kiosk to show accurate events, I need the calendar parser to correctly expand a monthly recurring event and to never crash on a recurrence pattern it doesn't yet support, so that one calendar's advanced RRULE shape can't take down that kid's (or the family's) entire blended event list per FR-19.

* Technical Notes

- *Evidence correction (recorded here since it shaped this story's scope):* the authorization for this story originally cited "7 MONTHLY/YEARLY recurring events (2 per kid calendar)" in the real feeds as an FR-19-blocking finding. Verified directly against =tmp/real-feeds/*.ics= before this story was written: scoped strictly to =VEVENT= blocks (excluding =VTIMEZONE='s boilerplate =DAYLIGHT=/=STANDARD= sub-blocks, which every Google export encodes as =RRULE:FREQ=YEARLY;BYMONTH=3;BYDAY=2SU= / =RRULE:FREQ=YEARLY;BYMONTH=11;BYDAY=1SU= DST-transition rules — not family events), the real feeds contain exactly *one* =FREQ=MONTHLY= =VEVENT= (in =child-b.ics=: =RRULE:FREQ=MONTHLY;WKST=SU;BYDAY=1TH= with two =EXDATE=s) and *zero* =FREQ=YEARLY= =VEVENT=s. The naive whole-file grep that produced "7" was counting the VTIMEZONE boilerplate lines (2 per file × 3 files = 6) plus the one real =MONTHLY= event = 7 — an artifact of not scoping the grep to =VEVENT= blocks, not seven distinct family events. This story is scoped to the corrected evidence: one real =MONTHLY= fixture, and =YEARLY= handled only via the graceful-degradation AC (no real =YEARLY= fixture exists or is required).
- Sanitization process (same mechanism as Story 01, not itself an AC): pull the family's real ICS feeds locally (never committed), hand-derive the fixture file by replacing =SUMMARY=/=LOCATION=/=ORGANIZER=/=ATTENDEE=/=UID= with placeholders while preserving the =RRULE=/=EXDATE=/=DTSTART=/=DTEND= structure the AC exercises, and verify with the same identifying-field grep Story 01 used for its sanitization AC. The real feeds and any intermediate un-sanitized files must never be staged or committed.
- AC4's "unsupported =FREQ=" test does not need a real fixture — construct a minimal =FREQ=YEARLY= ICS snippet inline in the test module, the same way Story 01's closeout audit reproduced the =CaseClauseError= (using =FREQ=MONTHLY=, before this story made it supported), mirroring Story 01's approach to the midnight-spanning fixture when no real analog existed.
- No dependency on Stories 02/03/04 — this story only extends =BearCub.Calendars.ICS= (=lib/bear_cub/calendars/ics.ex=, specifically =expand_rrule/3=); it does not touch the =calendars= schema/table, the =BearCub.Calendars= context, the Refresher GenServer, or any LiveView.
- Deferred discovery carried from Story 01, still open after this story: =BYMONTHDAY=, =BYSETPOS=, =BYMONTH=, =COUNT=, and =FREQ=YEARLY= expansion remain unimplemented — only =FREQ=MONTHLY;BYDAY=<ordinal><weekday>= (the shape the real feeds actually use) is added by this story. If a future real feed needs broader coverage, extend =expand_rrule/3= again rather than reaching for a library (D25 rules out =icalendar= categorically).
