* User Story

As a kid glancing at the fridge, I need to see today's events — mine and the family's — listed above my chores, so that I know what's happening today without anyone reading it to me; and as a parent, I need that events strip to keep working through a calendar outage so it never disrupts the chores my kid depends on.

* Acceptance Criteria

** [ ] Each kid's column shows a chronological list of today's events, blended from that kid's personal calendar(s) and the family calendar (FR-19).
** [ ] Family events are visually distinct from the kid's personal events — a neutral chip + house glyph, versus the kid-color dot used for personal events — and appear in both kids' columns.
** [ ] All-day events are pinned to the top of the events strip, ahead of timed events (FR-22).
** [ ] An event spanning midnight renders clipped to today's portion (e.g., "until 2:00 PM") rather than its full multi-day span.
** [ ] When any calendar's cache exceeds the staleness threshold, a small clock glyph appears alongside the existing socket-down indicator; it clears once every stale calendar has refreshed successfully (D4).
** [ ] A calendar refresh (admin edits a calendar, or the Refresher's background fetch changes events) appears on the kiosk without a manual reload, consistent with the existing live-update pattern (FR-9 extended to the ="calendars"= topic).
** [ ] With no calendars configured for a kid, that kid's events strip renders empty without error — chores render normally either way.
** [ ] LiveView integration tests cover the blended per-kid + family event list, all-day pinning, the midnight-spanning clip, the staleness glyph, and the empty-calendars case (Repo conventions: §11 testing expectations fold into ACs wherever a flow is touched).

* Technical Notes

- Depends on Story 03: the events strip renders via =Calendars.today_events(kid_id, today)= and subscribes to the ="calendars"= PubSub topic, both established there. Also depends on the =KioskLive= boundary timer and PubSub subscription pattern already in place from Phases 1–3 (git history; no stories exist for those phases per Repo conventions).
- The socket-down dim dot (also a D4 corner glyph) already exists from an earlier phase; this story adds only the staleness clock glyph alongside it, per the Design excerpt.
- FR-20's on-device AC ("Unplug the WAN, reboot the server → kiosk shows chores plus the last cached events") is a phase-close manual gate item per Repo conventions, not a story AC here — this story's "renders empty without error" and staleness-glyph ACs, plus Story 03's fetch-failure/boot-hydration coverage, are what's unit/integration-testable in CI.
- §1's contexts paragraph (DESIGN.org) applies here too: rendering calendar data must never make chore/completion rendering depend on =BearCub.Calendars= being healthy — a calendar exception must not break the chores half of the kiosk view.
