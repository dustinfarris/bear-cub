* User Story

As a parent relying on the kiosk to show accurate events, I need the calendar parser proven against real recurrence edge cases before any calendar UI is built, so that the kiosk never confidently shows a cancelled or un-moved event.

* Acceptance Criteria

** [ ] Sanitized ICS fixture files exist, derived from the family's real Google Calendar feeds, covering: a plain timed event, an all-day event, an event spanning midnight, a recurring event (RRULE), a cancelled recurring instance (EXDATE), and a moved/edited recurring instance (RECURRENCE-ID override).
** [ ] None of the committed fixture files contain real family names, event titles, locations, or any other identifying string from the family's actual calendars — every identifying string is a placeholder.
** [ ] A =BearCub.Calendars.ICS= module parses each fixture and expands recurring events into concrete instances, correctly honoring RRULE expansion, EXDATE exclusion, and RECURRENCE-ID override, verified by tests asserting the expected instances for each fixture.
** [ ] The parser implementation choice (the =icalendar= hex package, or a hand-rolled parser if it doesn't clear the correctness bar) is settled and recorded, based on its performance against these fixtures.

* Technical Notes

- This is the spike named in D16 — it runs before Story 02, 03, and 04, and before any calendar UI work, per the Design excerpt above.
- No dependency on prior batches: Phases 1–3 (chores/admin) are unrelated to this module. This story does not touch the =calendars= schema/table, the =BearCub.Calendars= context, the Refresher GenServer, or any LiveView — those land in Stories 02–04. This story's scope is =BearCub.Calendars.ICS= (the parsing module) plus its fixtures and tests.
- Sanitization process (not itself an AC, but the mechanism for meeting the ACs above): pull the family's real ICS feeds locally (never committed), then hand-derive fixture files by replacing SUMMARY/LOCATION/ORGANIZER/ATTENDEE and any other identifying fields with placeholders, while preserving the RRULE/EXDATE/RECURRENCE-ID/DTSTART/DTEND structure that the correctness bar exercises. The real feeds and any intermediate un-sanitized files must never be staged or committed.
- The Design excerpt's bounded horizon (yesterday → tomorrow) and ETS storage are called out here as context for why expansion must work correctly across a resolution window, but the ETS caching and the fetch/serve pipeline itself belong to Story 03 (§6 Fetch/Serve), not this story — this story proves the parsing/expansion logic on fixtures directly, independent of the fetch pipeline.
- =Calendars.ICS= is described in the design as "swappable behind the wrapper" — keep the module's public interface stable so Story 03 can call it without knowing which library backs it.
