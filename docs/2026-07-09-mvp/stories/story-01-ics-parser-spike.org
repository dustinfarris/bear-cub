As a parent relying on the kiosk to show accurate events, I need the calendar parser proven against real recurrence edge cases before any calendar UI is built, so that the kiosk never confidently shows a cancelled or un-moved event.

* Technical Notes

- This is the spike named in D16 — it runs before Story 02, 03, and 04, and before any calendar UI work, per the Design excerpt above.
- No dependency on prior batches: Phases 1–3 (chores/admin) are unrelated to this module. This story does not touch the =calendars= schema/table, the =BearCub.Calendars= context, the Refresher GenServer, or any LiveView — those land in Stories 02–04. This story's scope is =BearCub.Calendars.ICS= (the parsing module) plus its fixtures and tests.
- Sanitization process (not itself an AC, but the mechanism for meeting the ACs above): pull the family's real ICS feeds locally (never committed), then hand-derive fixture files by replacing SUMMARY/LOCATION/ORGANIZER/ATTENDEE and any other identifying fields with placeholders, while preserving the RRULE/EXDATE/RECURRENCE-ID/DTSTART/DTEND structure that the correctness bar exercises. The real feeds and any intermediate un-sanitized files must never be staged or committed.
- The Design excerpt's bounded horizon (yesterday → tomorrow) and ETS storage are called out here as context for why expansion must work correctly across a resolution window, but the ETS caching and the fetch/serve pipeline itself belong to Story 03 (§6 Fetch/Serve), not this story — this story proves the parsing/expansion logic on fixtures directly, independent of the fetch pipeline.
- =Calendars.ICS= is described in the design as "swappable behind the wrapper" — keep the module's public interface stable so Story 03 can call it without knowing which library backs it.

** Parser choice: icalendar hex package rejected, hand-rolled instead

=icalendar= 1.1.3 was added to =mix.exs=, fetched, and probed directly (=ICalendar.from_ics/1= and =ICalendar.Recurrence.get_recurrences/2=) against a two-=VEVENT= master+override pair (a =FREQ=WEEKLY= master plus a second =VEVENT= sharing its =UID= with a =RECURRENCE-ID=). Findings:

- *RECURRENCE-ID is silently dropped — hard fail against D16.* =ICalendar.Event= (=deps/icalendar/lib/icalendar/event.ex=) has no =recurrence_id= field, and =ICalendar.Util.Deserialize.parse_attr/2= has no clause for the =RECURRENCE-ID= key — it falls through to the catch-all clause that discards unrecognized properties. Expected: the override event's struct carries the value it overrides (=20260322T160000= in the probe) so callers can match it to the master's occurrence. Actual: the parsed override event is a plain =%ICalendar.Event{}= with an ordinary =dtstart=/=dtend=, indistinguishable from an unrelated one-off event that happens to share the same =UID=. There is no supported way to reconstruct which master occurrence it replaces, so a consuming pipeline built on this library cannot avoid showing both the un-moved original slot and the moved instance — exactly the failure D16 rules out.
- Secondary issues that would have needed work-arounds even for EXDATE/RRULE alone: =ICalendar.Recurrence= only supports the =:byday= by-x modifier (=@supported_by_x_rrules=); =BYMONTHDAY=, =BYSETPOS=, =BYMONTH=, =BYYEARDAY=, =BYWEEKNO= are explicitly unsupported per its own moduledoc. EXDATE exclusion is a raw =DateTime= equality check (=new_event.dtstart in new_event.exdates=), fragile across TZID/UTC representation mismatches. All-day (=VALUE=DATE=) events collapse to UTC-midnight timestamps with no =all_day= flag, so callers can't distinguish them from a real midnight timed event.
- Given the RECURRENCE-ID gap is a hard, structural fail (not a config/version issue — no field exists to populate), =BearCub.Calendars.ICS= was hand-rolled per the fixture-tested contract instead. The =icalendar= dependency was added, evaluated, and fully removed (=mix.exs=, =mix.lock=, =deps/=) — no trace remains in the committed tree.
- The hand-rolled implementation covers what the six required fixtures exercise: =FREQ=DAILY|WEEKLY= with optional =INTERVAL= and =BYDAY=, bounded by =UNTIL= and/or the caller's window (safety-capped at 1000 occurrences regardless, so an indefinite RRULE can't loop forever). =FREQ=MONTHLY|YEARLY=, =COUNT=, and the other =BYxxx= modifiers are unimplemented.
- Deferred discovery: broader RRULE coverage (=FREQ=MONTHLY|YEARLY=, =COUNT=, =BYMONTHDAY=, =BYSETPOS=, =BYMONTH=) — none of the six required fixtures need it, but the family's real feeds do contain a handful of =FREQ=DAILY= events with large =INTERVAL= values and no =COUNT=/=UNTIL=; if Story 03's real-feed fetch surfaces a recurrence shape this module can't expand, extend =BearCub.Calendars.ICS.expand_rrule/3= rather than reaching for a library, since the RECURRENCE-ID gap rules out =icalendar= categorically for this project.
- Deferred discovery: recurring *all-day* events (=RRULE= combined with =VALUE=DATE=) are unhandled — no required fixture combines them, and none were observed in the family's real feeds.
- Deferred discovery: =expand_rrule/3='s =case= has no catch-all — an unsupported =FREQ= (=MONTHLY=, =YEARLY=) raises =CaseClauseError= rather than degrading gracefully, crashing the whole =parse/3= call for that feed. Closeout audit flagged this; correctly out of this story's scope (=Calendars.ICS= proves parsing/expansion on fixtures directly, independent of the fetch pipeline per this story's own Technical Notes above). Story 03's fetch/serve pipeline should handle this defensively — e.g. rescue per-calendar rather than let one malformed feed take down the whole fetch — consistent with the "calendar is a degradable overlay" design invariant.

** Fixture provenance note: midnight-spanning event

None of the three real feeds (=family.ics=, =child-a.ics=, =child-b.ics=) contain an event whose *local* (=America/Los_Angeles=) wall-clock time spans midnight — the family's evening events are either encoded with a local =TZID= that stays within one local calendar day, or as UTC (=Z=) instants that cross a UTC-day boundary while remaining same-day in Pacific time. =test/support/fixtures/ics/midnight_spanning_event.ics= therefore is *not* a sanitized excerpt of a real event: its =DTSTART=/=DTEND= wall-clock times (22:30 → 00:30 =America/Los_Angeles=) were constructed to exercise this correctness-bar case, using the same structural shape (=DTSTART;TZID=America/Los_Angeles=, single =VEVENT=, no =RRULE=) as the family's real single-instance events. Every other fixture's =RRULE=/=EXDATE=/=RECURRENCE-ID=/=DTSTART=/=DTEND= structure is a direct, only-SUMMARY/LOCATION/UID-sanitized excerpt of a real =VEVENT= (or =VEVENT= pair) from =family.ics=.
